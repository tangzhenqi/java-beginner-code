# 注解 + 反射：手写一个极简 JUnit

本例用「自定义注解 + 反射」实现了一个极简版测试运行器：**只运行被 `@MyTest` 标记的方法**。这正是 JUnit、Spring 等注解框架的底层原理。

涉及三个文件：

| 文件 | 角色 |
| --- | --- |
| `MyTest.java` | 自定义注解（标签本身） |
| `CalculatorTests.java` | 被测类（往方法上贴标签） |
| `TestRunner.java` | 运行器（读标签并按标签行动） |

---

## 一、核心思想：注解只是"标记"，不带行为

注解（`@interface`）本身**不会做任何事**。`@MyTest` 贴在方法上，只是在字节码里留了一个"标签/元数据"。

真正让它"生效"的，是**别人用反射去读这个标签，再决定做什么**。

> 一句话：**注解 = 元数据（标签），反射 = 读标签并行动。注解本身不做事，是"读它的代码"赋予了它意义。**

---

## 二、第一步：定义注解

```java
@Retention(RetentionPolicy.RUNTIME)   // 关键：运行期保留，反射才读得到
@Target(ElementType.METHOD)           // 只能贴在方法上
public @interface MyTest {
    String value() default "";        // 注解携带的属性
}
```

### 两个元注解

| 元注解 | 作用 |
| --- | --- |
| `@Retention` | 注解保留到哪个阶段 |
| `@Target` | 注解能贴在什么位置（类/方法/字段…） |

### @Retention 的三种策略（重点）

| 策略 | 保留到 | 反射能否读到 | 举例 |
| --- | --- | --- | --- |
| `SOURCE` | 仅源码，编译后丢弃 | ❌ | `@Override`、`@SuppressWarnings` |
| `CLASS` | 编译进 class，类加载时丢弃 | ❌ | 默认值 |
| `RUNTIME` | 保留到运行期 | ✅ | 本例 `@MyTest` |

**只有 `RUNTIME` 的注解，反射才读得到。** 这是整套机制能工作的前提。

---

## 三、第二步：在目标上贴注解

```java
public class CalculatorTests {

    @MyTest("加法应当正确")
    public void testAdd() {
        if (1 + 2 != 3) throw new AssertionError("1+2 != 3");
    }

    @MyTest("演示一个会失败的用例")
    public void testFail() {
        throw new IllegalStateException("故意失败用来看效果");
    }

    // 没贴注解 —— 应该被跳过
    public void helperShouldBeIgnored() {
        System.out.println("不应当被调用！");
    }
}
```

贴与不贴，决定了后面运行器会不会执行它。

---

## 四、第三步：反射读标签并执行（注解真正"生效"处）

```java
Object instance = testClass.getDeclaredConstructor().newInstance();   // ① 创建被测类实例

for (Method m : testClass.getDeclaredMethods()) {                     // ② 遍历所有方法
    if (!m.isAnnotationPresent(MyTest.class)) {                       // ③ 有没有贴 @MyTest？
        continue;                                                    //    没有就跳过
    }
    MyTest meta = m.getAnnotation(MyTest.class);                      // ④ 读注解携带的数据
    String name = m.getName() + "（" + meta.value() + "）";           //    meta.value() = "加法应当正确"
    try {
        m.setAccessible(true);
        m.invoke(instance);                                          // ⑤ 按标签执行该方法
        System.out.println("[PASS] " + name);
    } catch (InvocationTargetException e) {
        Throwable cause = e.getCause();                              // ⑥ 剥壳拿真实异常
        System.out.println("[FAIL] " + name + " -> "
                + cause.getClass().getSimpleName() + ": " + cause.getMessage());
    }
}
```

### 逐步解释

| 步骤 | API | 作用 |
| --- | --- | --- |
| ① | `getDeclaredConstructor().newInstance()` | 反射拿无参构造，创建实例（下面要调实例方法，得先有对象） |
| ② | `getDeclaredMethods()` | 拿到该类声明的**所有**方法（含没注解的） |
| ③ | `isAnnotationPresent(MyTest.class)` | **判断方法上有没有 `@MyTest`**，没有就跳过 → 过滤掉 `helperShouldBeIgnored` |
| ④ | `getAnnotation(MyTest.class)` | **拿到注解对象**，`meta.value()` 读出属性值 → 注解能携带配置数据 |
| ⑤ | `setAccessible(true)` + `invoke(instance)` | **反射调用被标记的方法**；`setAccessible` 关闭访问检查，private 也能调 |
| ⑥ | `e.getCause()` | 见下方"异常剥壳" |

### 异常剥壳（易错点）

反射调用时，被调方法内部抛出的异常，会被 `invoke` **统一包装成 `InvocationTargetException`**。
真正的业务异常（如 `testFail` 里的 `IllegalStateException`）藏在 `e.getCause()` 里，要剥一层才能拿到真实原因。

---

## 五、整体套路

```
定义注解(@interface + RUNTIME)
        │
        ▼
在目标上贴注解(@MyTest)
        │
        ▼
反射扫描
   ├─ isAnnotationPresent  判断有没有贴
   ├─ getAnnotation        读注解携带的数据
   └─ invoke               据此执行相应逻辑
```

## 六、运行结果

| 方法 | 结果 |
| --- | --- |
| `testAdd` | `[PASS] testAdd（加法应当正确）` |
| `testFail` | `[FAIL] testFail（演示一个会失败的用例）-> IllegalStateException: 故意失败用来看效果` |
| `helperShouldBeIgnored` | 未贴注解，被跳过，不打印 |
| 统计 | `总计: 2 通过: 1 失败: 1` |

## 七、小结

- 注解 = **元数据（标签）**，本身不执行任何逻辑。
- 注解要被反射读到，必须是 **`@Retention(RUNTIME)`**。
- "实现注解"三板斧：
  - `isAnnotationPresent(...)` —— 判断有没有贴
  - `getAnnotation(...)` —— 读注解里的属性
  - `invoke(...)` —— 据此执行
- 反射 `invoke` 抛出的异常要用 `getCause()` **剥壳**取真实原因。
- JUnit、Spring 的 `@Component`、`@Autowired`、`@Transactional` 等全都是这套机制：**框架启动时用反射扫描注解，再决定注入/执行/拦截什么。**
