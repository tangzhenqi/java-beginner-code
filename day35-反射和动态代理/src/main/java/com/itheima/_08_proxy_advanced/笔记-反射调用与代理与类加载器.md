## 一、反射调用方法：`Method.invoke`

```java
Person target = new Person();
Object ret = m.invoke(target, "汉堡包");   // 等价于 target.eat("汉堡包")
```

`Object invoke(Object obj, Object... args)` 的三要素：

- **参数一 `obj`**：用哪个对象去调。实例方法必须传对象；**静态方法传 `null`**。
- **参数二 `args`**：方法实参，按形参顺序传。
- **返回值**：方法本身的返回值，统一以 `Object` 接收（基本类型会自动装箱）；方法为 `void` 时返回 `null`。

要点：
- 调 `private` 方法前需 `m.setAccessible(true)`（关闭访问检查，俗称暴力反射），否则抛 `IllegalAccessException`。
- 方法内部抛出的异常会被包装成 `InvocationTargetException`，真正的异常在 `getCause()` 里。

| 普通调用 | 反射调用 |
|---|---|
| `target.eat("汉堡包")` | `m.invoke(target, "汉堡包")` |
| 编译期确定调哪个方法 | 运行时由 `Method` 对象决定 |
| 受 `private` 限制 | 可用 `setAccessible(true)` 绕过 |
| 返回具体类型 | 返回 `Object`，可能需强转 |

反射的价值：调用哪个对象、哪个方法、传什么参数都能**运行时动态决定**，是 Spring、MyBatis 等框架的底层基石。

## 二、JDK 动态代理参数：`Proxy.newProxyInstance`

```java
Proxy.newProxyInstance(
        target.getClass().getClassLoader(),   // ① 类加载器
        target.getClass().getInterfaces(),     // ② 要实现的接口
        (proxy, method, args) -> { ... });     // ③ InvocationHandler
```

原理：JDK 在运行时**凭空生成一个新类**（如 `$Proxy0`）并 new 它的实例返回。生成这个类必须知道两件事：

- **① `getClassLoader()`**：用哪个类加载器加载生成的代理类。复用 target 的加载器，保证代理类和被代理类处在**同一加载环境**，能互相认识、类型转换不出错。
- **② `getInterfaces()`**：代理类要实现哪些接口，决定它"能被当成什么用"。传 target 实现的接口，使代理对象对外暴露同一套接口，可强转成 `T` 使用。
- **③ `InvocationHandler`**：代理对象每个方法调用都转发到这里统一处理（加日志等）。

固有限制：
- **JDK 动态代理只能代理接口**。`getInterfaces()` 为空则代理无意义。
- `getInterfaces()` 只返回**直接实现**的接口，不含从父类继承来的接口。

## 三、CGLIB 代理

与 JDK 代理最大区别：**JDK 基于「接口」，CGLIB 基于「继承」**——运行时生成被代理类的**子类**，重写方法插入逻辑。**不要求 target 实现接口**。

依赖（标准库没有）：
```xml
<dependency>
    <groupId>cglib</groupId>
    <artifactId>cglib</artifactId>
    <version>3.3.0</version>
</dependency>
```

示例（对照 `LogProxyFactory`）：
```java
public static <T> T wrap(T target) {
    Enhancer enhancer = new Enhancer();
    enhancer.setSuperclass(target.getClass());          // ① 指定父类
    enhancer.setCallback((MethodInterceptor) (obj, method, args, proxy) -> {
        System.out.println("[LOG] -> " + method.getName());
        Object ret = proxy.invokeSuper(obj, args);      // ② 调父类原方法
        System.out.println("[LOG] <- " + method.getName() + " return=" + ret);
        return ret;
    });
    return (T) enhancer.create();                       // ③ 生成代理实例
}
```

| | JDK 动态代理 | CGLIB |
|---|---|---|
| 生成方式 | 实现接口 | 继承 target 的类 |
| 前置条件 | 必须有接口 | 无需接口，但类/方法不能 `final` |
| 入口 API | `Proxy.newProxyInstance` | `Enhancer` |
| 拦截回调 | `InvocationHandler` | `MethodInterceptor` |
| 调原方法 | `method.invoke(target, args)` | `proxy.invokeSuper(obj, args)` |

两个坑：
1. CGLIB 里调原逻辑用 `proxy.invokeSuper(obj, args)`，**不能用 `method.invoke(obj, args)`**，否则无限递归 `StackOverflowError`。
2. 靠继承+重写，所以 `final` 类不能代理，`final`/`private`/`static` 方法无法拦截。

环境提示（JDK 11）：CGLIB 用了 `sun.misc.Unsafe` 等内部 API，JDK 9+ 可能需加
`--add-opens java.base/java.lang=ALL-UNNAMED`。Spring 现用内置 fork 的 `org.springframework.cglib` + ByteBuddy。

## 四、类加载器（ClassLoader）

作用：把 `.class` 字节码加载进内存、变成 `Class` 对象的"搬运工"。`Class.forName(...)` 就是触发它去加载类。

类加载三步：
1. **加载 Loading**：找到 `.class`，读成字节流，生成 `Class` 对象。
2. **链接 Linking**：验证字节码、为静态变量分配内存赋默认值、解析符号引用。
3. **初始化 Initialization**：执行静态代码块、给静态变量赋真正初值。

三层加载器：

| 加载器 | 负责加载 |
|---|---|
| Bootstrap（启动） | JDK 核心类 `java.lang.*`（`java.base`）。C++ 实现，Java 里拿到是 `null` |
| Platform/Extension（平台/扩展） | JDK 扩展库 |
| Application（应用） | 自己写的类、classpath 上的 jar。**最常用** |

**双亲委派机制**：加载类时先交给父加载器尝试，一层层往上到 Bootstrap；上面加载不了才自己来。
```
Application → Platform → Bootstrap（先试）
```
目的：保证核心类唯一和安全。自己写的 `java.lang.String` 会被委派上去，返回官方的，伪造版永远加载不了，防止核心类被篡改与重复加载。

**类的身份**：JVM 判断两个类是否同一个，不只看全限定名，还看**加载它的类加载器**。同一个 `.class` 被两个不同加载器加载，会得到两个互不相等、不能强转的 `Class`（强转抛 `ClassCastException`）。Tomcat 每个 web 应用独立加载器、热部署替换类，靠的就是这个特性。

与代理的关系：动态代理运行时生成的 `$Proxy0` 不存在于磁盘，必须指定一个类加载器塞进 JVM——这就是 `newProxyInstance` 第一个参数传 target 加载器的原因。
