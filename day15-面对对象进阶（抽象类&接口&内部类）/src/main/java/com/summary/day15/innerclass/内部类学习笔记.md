# Java 内部类学习笔记

> 整理自 day15 内部类示例的学习讨论，重点厘清"内部类到底有没有用、什么时候用哪种"。

## 一、核心结论：内部类难读？你的直觉是对的

"用了内部类可读性变差"——这个吐槽主要针对**成员内部类（非静态）**，它确实是被滥用、最难读的一种。但"内部类"是个大类别，四种实用度天差地别：

| 类型 | 实用度 | 说明 |
|------|--------|------|
| 成员内部类（非静态） | ⭐ 很少用 | 难读的元凶，隐式持有外部对象引用 |
| 静态内部类 | ⭐⭐⭐⭐ 常用 | 实际开发主力 |
| 局部内部类 | ⭐ 几乎不用 | 被 lambda 取代 |
| 匿名内部类 | ⭐⭐⭐⭐ 常用 | 单方法接口可被 lambda 简化 |

**学习建议**：重点掌握「静态内部类」和「匿名内部类」；成员内部类、局部内部类看得懂即可。

## 二、四种内部类对照

### 1. 成员内部类（非静态）
- 必须依附外部类对象才能创建：`new Car("宾利").new Engine("V12")`
- 隐式持有外部对象引用，可直接访问外部类**所有**成员（含 private）
- 同名变量区分：局部变量 → `x`；内部类成员 → `this.x`；外部类成员 → `外部类名.this.x`
- 风险：白白持有外部引用，Android 里典型内存泄漏来源

### 2. 静态内部类（主力）
- 创建**不需要、也不会**产生外部类对象：`new Holder.Box(42)` 只创建 Box
- 本质是独立类，只是"住"在外部类里表达归属关系
- 只能直接访问外部类的**静态**成员；要访问非静态成员需自己 `new` 外部对象
- 典型应用：`Map.Entry`、`HashMap.Node`、Builder 模式

### 3. 局部内部类
- 定义在方法内部，作用域仅限该方法
- 可访问所在方法的局部变量，但这些变量必须是 `final` 或 **effectively final**（事实不可变）
- 原因：内部类捕获的是变量拷贝，强制不可变以避免"看似同步实则不同步"的 bug
- 实际开发几乎不用，被 lambda 取代

### 4. 匿名内部类（常用）
- 格式：`new 接口/父类() { 重写方法 }`，本质是一个"没名字的子类对象"
- 常用于：方法实参临时传实现、事件回调
- **能否用 lambda 简化**：仅当父类型是「函数式接口」（只有一个抽象方法）时可以；抽象类不行

```java
// 函数式接口 → 可简化
Swimmer s = () -> System.out.println("仰泳");
// 抽象类 → 只能用匿名内部类
Mascot m = new Mascot("冰墩墩") { public void perform(){...} };
```

## 三、静态 vs 非静态：一句话判断法

> **内部类需要访问"外部类某个具体对象"的非静态字段吗？**
> - **不需要** → 用 `static`（轻量、可独立创建、不持有外部引用）
> - **需要** → 去掉 `static`（依附外部对象）

**实操经验法则**：默认先写 `static`；只有当编译器报错"无法从静态上下文访问外部实例字段"时，再去掉 `static`。这也是 JDK 源码一贯做法。

### 对比示例（见 `WhenToUseStaticInnerClassDemo.java`）

| 类 | 写法 | 结论 |
|----|------|------|
| `GoodCart.Item` | 静态内部类 ✅ | Item 是纯数据，不关心属于哪个 Cart |
| `BadCart.Item` | 成员内部类 ❌ | 纯数据却被迫绑定实例，白持引用、创建别扭 |
| `MyList.Iterator` | 成员内部类 ✅ | 迭代器必须知道遍历哪个具体列表、要读外部 `data` 字段 |
