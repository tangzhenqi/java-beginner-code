# 接口静态方法 / 工厂方法 / 函数式接口 学习笔记

> 配套代码：`day15/interfacekw/_05_StaticMethodDemo.java`

## 一、起点：这几行代码是什么意思

```java
// 工厂方法：通过接口给出一个默认实现
MathOps ops = MathOps.defaultOps();
System.out.println("默认实现 * 2 = " + ops.times(2));
```

对应的接口定义：

```java
interface MathOps {
    int times(int x);                 // 唯一的抽象方法

    static int add(int a, int b) {    // 静态工具方法
        return a + b;
    }

    static MathOps defaultOps() {     // 静态工厂方法
        return x -> x * 10;           // 用 lambda 返回一个实现对象
    }
}
```

- `MathOps.defaultOps()` 调用接口的**静态方法**，返回值赋给 `MathOps` 类型变量 `ops`。
- 返回的 `x -> x * 10` 是一个 lambda，实现了唯一抽象方法 `times`。
- 所以 `ops.times(2)` = `2 * 10 = 20`，打印 `默认实现 * 2 = 20`。

---

## 二、什么是「工厂方法」

工厂方法是个比喻：**不直接 `new` 对象，而是调用一个方法，让它替你造好对象再交给你。**

```java
MathOps ops = new MathOpsImpl();      // 普通做法：自己 new，必须先有实现类
MathOps ops = MathOps.defaultOps();   // 工厂做法：调方法，内部帮你造好
```

核心好处：**把「怎么造」和「怎么用」分开**，调用者不关心具体实现类。

---

## 三、为什么「省了一个实现类」

### 传统写法：必须单独建实现类

```java
interface MathOps { int times(int x); }

class DefaultMathOps implements MathOps {     // 被迫多写一整个类
    @Override
    public int times(int x) { return x * 10; }
}

MathOps ops = new DefaultMathOps();
```

### JDK 8 写法：接口静态方法 + lambda

```java
interface MathOps {
    int times(int x);
    static MathOps defaultOps() {
        return x -> x * 10;           // 一行 lambda 等价于上面整个类
    }
}

MathOps ops = MathOps.defaultOps();
```

编译器在背后帮你生成了一个匿名实现对象，**不必再额外建 `XxxImpl` 文件**。

---

## 四、JDK 8 接口静态方法的典型用途

| 用途 | 含义 | 本例对应 |
|------|------|----------|
| **工厂方法** | 帮你造好并返回本接口的实现对象 | `defaultOps()` |
| **工具方法** | 与本接口相关、不需要对象就能用的辅助计算 | `add(a, b)` |

规则要点：
- 静态方法**只能通过「接口名」调用**，不能用实现类名 / 对象名调用。
- `public` 可省略，`static` 不能省略。
- 实现类中同名同参的 static 方法**不是重写**，互不干扰。

> JDK 8 之前这类方法只能放在单独的 `XxxUtils` 工具类里，导致「接口和它的工具被拆成两处」。JDK 8 允许接口写静态方法后，可把工厂/工具方法收进接口本身（如 `Comparator.naturalOrder()`、`Stream.of(...)`）。

## 五、关键追问：如果接口有多个抽象方法呢？

> **结论：lambda 只能用在「只有一个抽象方法」的接口上。多个抽象方法时 lambda 不能用，会编译报错。**

### 为什么

lambda `x -> x * 10` **没有写方法名**。若接口只有一个抽象方法，没有歧义；若有多个：

```java
interface MathOps {
    int times(int x);
    int plus(int x);   // 多了一个
}
```

编译器无法判断 lambda 实现的是 `times` 还是 `plus`，直接报错：
`MathOps 不是函数接口 / 找到多个非覆盖抽象方法`。

### 函数式接口（Functional Interface）

**「有且只有一个抽象方法」的接口叫函数式接口，只有它才能用 lambda。**

```java
@FunctionalInterface   // 编译器检查：必须正好一个抽象方法，否则报错
interface MathOps {
    int times(int x);
}
```

> 注意：`default` 方法、`static` 方法**不算**抽象方法，有几个都不影响。只数「抽象方法」的个数。所以本例中 `add`、`defaultOps` 是 static，`MathOps` 仍是函数式接口。

### 多个抽象方法时怎么写实现

**方式 1：匿名内部类**

```java
MathOps ops = new MathOps() {
    @Override public int times(int x) { return x * 10; }
    @Override public int plus(int x)  { return x + 10; }
};
```

**方式 2：单独建实现类**

```java
class MyMathOps implements MathOps {
    @Override public int times(int x) { return x * 10; }
    @Override public int plus(int x)  { return x + 10; }
}
MathOps ops = new MyMathOps();
```

### 对照表

| 抽象方法个数 | 能用 lambda 吗 | 怎么写实现 |
|---|---|---|
| 1 个（函数式接口） | ✅ 能 | `x -> x * 10` 简写 |
| 2 个或更多 | ❌ 不能（编译报错） | 匿名内部类 / 独立实现类 |
| 1 个抽象 + 多个 default/static | ✅ 能 | default/static 不计入 |

---

## 六、一句话总结

- **lambda** 是「单方法接口」的语法糖，省略了方法名，所以接口必须**只有一个抽象方法**才无歧义。
- **接口静态方法**（如 `defaultOps`）给「造对象」一个固定入口，用 `接口名.方法名()` 调用。
- 两者配合 → **接口自己成了自己的工厂**，一句 `MathOps.defaultOps()` 就拿到默认实现，干净又集中。
- 多个抽象方法时用不了 lambda，回到匿名内部类或独立实现类逐个实现；可用 `@FunctionalInterface` 锁定接口只能有一个抽象方法。
