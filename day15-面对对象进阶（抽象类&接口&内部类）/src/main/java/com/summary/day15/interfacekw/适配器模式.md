# 适配器模式（Adapter Pattern）总结

> 配套代码：`src/main/java/com/summary/day15/interfacekw/_07_AdapterPatternDemo.java`

## 一、解决什么痛点

接口里往往有很多抽象方法，但实现类常常只关心其中一两个。
直接 `implements` 接口，会被迫把**所有方法**都写一遍（哪怕是空方法体），非常啰嗦。

```java
class MyAppListener implements WindowListener {
    public void onOpen()     { }   // 不关心，但必须写
    public void onClose()    { System.out.println("清理"); }
    public void onResize()   { }   // 不关心，但必须写
    public void onMove()     { }   // 不关心，但必须写
    public void onMinimize() { }   // 不关心，但必须写
}
```

## 二、解决思路

1. 保留原接口（定义契约）。
2. 额外写一个**抽象适配器类**，对接口的**所有方法**提供空实现。
3. 业务类去 `extends` 这个适配器，**只重写自己关心的方法**即可。

```java
/** 监听接口：定义"必须能响应哪些事件" */
interface WindowListener {
    void onOpen();
    void onClose();
    void onResize();
    void onMove();
    void onMinimize();
}

/** 抽象适配器：对每个方法提供空实现，写成 abstract 避免被直接 new */
abstract class WindowAdapter implements WindowListener {
    @Override public void onOpen()     { }
    @Override public void onClose()    { }
    @Override public void onResize()   { }
    @Override public void onMove()     { }
    @Override public void onMinimize() { }
}

/** 业务类只重写关心的方法，其余保持默认空实现 */
class MyAppListener extends WindowAdapter {
    @Override public void onClose()  { System.out.println("窗口关闭，做清理"); }
    @Override public void onResize() { System.out.println("窗口改变，重新布局"); }
}
```

## 三、接口和适配器的职责区别（核心理解）

很多人会觉得"有了适配器抽象类，为什么还要接口"，其实两者职责完全不同：

| | 角色 | 给谁用 |
|---|---|---|
| `interface WindowListener` | 定义"必须能干什么"的**契约** | 框架 / 调用方 |
| `abstract WindowAdapter` | 提供空实现，省去写废方法的**便利** | 业务类（图省事） |

- **接口管"对外的约定"**：框架只认接口类型，例如 `addWindowListener(WindowListener l)`，框架不知道也不该知道适配器的存在。
- **适配器管"对内的便利"**：纯粹是为了少写空方法。

### 接口为什么不能省

1. **Java 单继承限制**：业务类一旦 `extends WindowAdapter` 就不能再继承别的类；而接口可以实现多个，更灵活。
2. **解耦**：框架依赖接口而不是某个具体空实现类，不会被绑死。

## 四、JDK 中的真实例子

`WindowAdapter`、`MouseAdapter`、`KeyAdapter` 等（AWT / Swing 事件监听体系）。

## 五、与 JDK 8 default 方法的关系（重要）

适配器模式**本质上是 JDK 8 之前的产物**。那时接口的方法只能是抽象的，无法提供默认实现，只能靠一个抽象类来"垫"空实现。

JDK 8 之后，可以直接在接口里用 `default` 方法提供空实现，从而**省掉适配器类**：

```java
interface WindowListener {
    default void onOpen()     { }
    default void onClose()    { }
    default void onResize()   { }
    default void onMove()     { }
    default void onMinimize() { }
}
```

### 那还要不要学适配器？要，原因：

- 老代码、JDK 自带的 AWT/Swing 里大量使用，必须看得懂。
- 语义差别：`default` 方法是接口"对外承诺"的一部分，会被所有实现类继承；适配器的空实现属于"内部约定"，可随意修改而不影响接口契约。

## 六、一句话总结

接口管**对外的约定**，适配器管**对内的便利**，所以不是重复。
只不过 JDK 8 之后，适配器提供的便利可以用接口 `default` 方法平替掉。
