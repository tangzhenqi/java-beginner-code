# 抽象类为什么不能直接 new？底层原理

> 本文总结自一次问答：抽象类不能 `new`，本质上是 **Java 语言规范的规定 + JVM 字节码层面的强制检查**。下面分编译期、运行期、设计原因三层来讲。

---

## 一、编译期：编译器直接拦截

当你写 `new AbstractBasicsDemo()` 时，`javac` 会检查目标类的访问标志。如果类带有 `abstract` 修饰符，编译器直接报错：

```
error: AbstractBasicsDemo is abstract; cannot be instantiated
```

这是 JLS（Java 语言规范 §15.9.1）明确规定的：类实例创建表达式的目标类型不能是抽象类。所以大部分情况下你根本过不了编译这一关。

---

## 二、运行期：JVM 字节码层面的强制

就算绕过编译器（比如用反射、字节码生成），JVM 也会拦你。原因在于"创建对象"这件事在字节码里其实是 **两步**：

```
new AbstractBasicsDemo()   →   编译成两条指令
```

| 字节码指令 | 作用 |
|-----------|------|
| `new` | 在堆上分配内存，得到一个未初始化的对象引用 |
| `invokespecial <init>` | 调用构造方法，完成初始化 |

关键点：**`new` 指令本身会做校验**。JVM 规范规定，`new` 指令的操作数如果指向一个抽象类（或接口），会抛出：

```
InstantiationError
```

即使你用反射 `clazz.newInstance()` / `Constructor.newInstance()`，底层最终也要执行类似 `new` 的分配 + 构造逻辑，同样会抛 `InstantiationException` / `InstantiationError`。

---

## 三、为什么语言要这么设计（根本原因）

抽象类的存在意义就是 **"不完整"**：

1. **可能有抽象方法没有方法体**。如果允许 `new`，那调用这个没有实现的抽象方法时，JVM 根本不知道执行哪段代码——没有任何指令可以跳转。
2. **它表达的是一种"概念/模板"，不是具体实体**。比如"动物"是抽象的，真正存在的是"猫""狗"。语言通过禁止实例化，从语法上强制你必须先提供一个具体的子类。

> 注意：抽象类 **有构造方法**，只是它不能被 `new` 直接调用，而是在子类对象创建时，通过 `super()` 被子类构造方法调用——用于初始化抽象类里定义的那部分字段。这也说明"抽象类不能 new"≠"抽象类没有构造器"。

---

## 四、一句话总结

> 抽象类不能 new，表层是 **编译器按 `abstract` 标志拒绝**，底层是 **JVM 执行 `new` 指令时检测到抽象类抛 `InstantiationError`**；而根本原因是抽象类可能含未实现的方法，它在设计上就是"不完整的模板"，必须由具体子类补全后才能实例化。
