# Java vs C++ 对比：虚方法表、类型指针与对象生命周期

> 本文集中收录 Java 与 C++ 在虚方法分派、类型指针、构造/析构期行为、对象销毁等方面的逐项对比。
> 各语言的纯讲解见《[[Java 虚方法表与对象底层机制]]》与《[[C++ 虚函数与对象底层机制]]》。

---

## 一、虚方法表 (vtable / itable)

| 对比项 | C++ | Java |
|--------|-----|------|
| 默认是否虚方法 | 否，需 `virtual` 关键字 | 是，除非 `static/private/final` |
| vtable 存放位置 | 每个对象首部存 vptr | 对象存 Klass Pointer，vtable 在类元数据中 |
| 多重继承 | 复杂，可能有多个 vptr | 不支持类多继承，只有接口（用 itable） |
| 调用开销 | 一次间接寻址 | 一次间接寻址（类似），但 JIT 可激进内联 |
| 运行时优化 | 编译期固定 | JIT 有**类型剖析 + 去虚化 + 内联** |
| vtable 何时生成 | 编译期静态构造，放入 `.rodata` | 类加载时构建于元空间的 InstanceKlass |

**关键差异**：C++ 默认静态绑定、需显式 `virtual`；Java 默认虚方法。Java 的多态分派还能被 JIT 在运行期"去虚化"为直接调用 + 内联，这是 Java 虚调用有时能跑赢 C++ 的原因之一。

## 二、类型指针的设置时机

| 时机 | Java (Klass Pointer) | C++ (vptr) |
|------|---------------------|-----------|
| 内存分配后 | 立即设置为最终类型 (Child) | 暂时未设置 |
| 进入父类构造器 | 仍是 Child | 设为父类 vtable |
| 进入子类构造器 | 仍是 Child | 改为子类 vtable |
| 构造结束 | 仍是 Child | 仍是 Child |

| 问题 | Java | C++ |
|------|------|-----|
| **类型指针何时被设置** | `new` 指令中，构造函数之前 | 构造函数序言中，用户代码之前 |
| **构造函数是否负责设置** | ❌ 由 `new` 负责 | ✅ 由编译器插入的序言负责 |
| **构造期间是否变化** | ❌ 始终是最终类型 | ✅ 跟随构造层次变化 |
| **构造期间 getClass / typeid** | 始终返回最终类型 | 返回当前正在构造的类 |
| **底层证据** | 字节码：`new` 与 `<init>` 是两条独立指令 | 编译器为每个构造器生成 vptr 赋值的序言代码 |

**为什么时机不同？**

- **Java**：类型指针是对象身份的一部分，必须立即确定——因为 GC、反射、`getClass()`、`instanceof` 在任何时刻都依赖它。所以"对象一诞生（`new`），类型就锁定"。
- **C++**：vptr 仅服务于虚函数分派，按需调整——没有 GC，元信息编译期固定，构造期间子类还不完整，调用子类虚函数会出错，所以 vptr 跟随"当前正在构造的层次"动态变化。

## 三、构造函数中能否触发多态

**简短答案：Java 可以（但有陷阱），C++ 不可以（被语言强制禁止）。**

| 维度 | Java 的选择 | C++ 的选择 |
|------|------------|----------|
| **vptr / Klass Pointer 是否可变** | 不变，构造时就是最终类型 | 在构造/析构各阶段动态调整 |
| **构造期调用虚方法** | 走子类版本（容易出 bug）| 走当前正在构造的类版本（安全）|
| **设计哲学** | 简化对象模型，把责任交给程序员 | 语言层面消除错误，性能换正确性 |
| **代价** | 程序员要自己避免陷阱 | 失去构造期多态能力 |

| 关键点 | Java | C++ |
|--------|------|-----|
| **能否触发多态** | ✅ 能，但会调用到字段未初始化的子类方法 | ❌ 不能，vptr 在构造期指向当前类 |
| **机制** | Klass Pointer 一直指向最终类型 | vptr 在每个构造器入口被改写 |
| **风险** | 程序员要自觉规避 | 语言强制不出错 |
| **最佳实践** | 构造器只调用 `final`/`private` 方法 | 自然安全，无需特别注意 |

**一句话**：
- **C++** 选择"在构造期间假装没有多态"——**安全但失去能力**
- **Java** 选择"全程保持多态"——**强大但留下陷阱**

> 其它语言：Kotlin 和 Java 同样有此陷阱，但通过 `open` 关键字让"可重写"显式化；Scala 行为同 Java；C# 同样有此陷阱，编译器警告 `CA2214: Do not call overridable methods in constructors`。

## 四、析构 / 终结期的类型指针

| 维度 | C++（析构函数） | Java（终结过程） |
|------|---------------|---------------|
| **触发时机** | 确定（作用域结束 / `delete`） | 不确定（GC 决定） |
| **是否有继承链调用** | ✅ 子→父，编译器生成 | ❌ `finalize` 需手动 `super.finalize()` |
| **类型指针状态** | 跟随当前正在析构的层次变化 | 始终是最终类型 |
| **进入子类析构** | vptr → 子类 | Klass Pointer → 子类 |
| **进入父类析构** | vptr **回退**到父类 | （无此阶段） |
| **析构期虚调用** | 走"当前层"的实现 | 走子类实现（如果在 `finalize` 中调用其它虚方法） |
| **谁改写指针** | 编译器插入的尾声代码 | 不改写 |
| **指针何时失效** | 内存释放那一刻 | 内存被 GC 回收那一刻 |

### 与构造期完全对称的"指针升降图"

```
       构造期                  使用期               析构期
C++:   vptr 逐层升级 ────────► vptr 稳定 ────────► vptr 逐层回退
       Parent → Child            Child              Child → Parent

Java:  Klass 一次到位 ────────► Klass 稳定 ────────► Klass 始终不变
       Child                     Child              Child
```

**一句话总结**：C++ 的 vptr 像"在楼梯上行走"——构造时一层层上楼，析构时一层层下楼；Java 的 Klass Pointer 像"贴在身上的身份证"——一旦发了就跟着对象到死。

## 五、对象销毁机制

| 维度 | C++ | Java |
|------|-----|------|
| 销毁机制 | 析构函数（确定性）| GC 回收（非确定性）|
| 触发时机 | 作用域结束 / `delete` | 不可控，由 GC 决定 |
| 谁负责调用 | 编译器插入 / 程序员 | JVM 垃圾回收器 |
| 内存释放 | `operator delete` → `free` | GC 整片回收 / 复制 |
| 循环引用 | 引用计数无法处理（需 weak_ptr）| 可达性分析天然解决 |
| 资源释放（非内存）| RAII，析构中确定释放 | try-with-resources / Cleaner |
| 异常安全 | 栈展开逆序析构 | finally / try-with-resources |
| 单对象销毁成本 | 每个都要析构 + free | 死对象可能零成本 |
| 析构顺序保证 | 严格逆序（成员、基类）| 无顺序保证 |
| 是否有析构函数 | ✅ `~ClassName()` | ❌ 只有 `finalize()`（已废弃）/ `Cleaner` |

## 六、最深层的哲学差异

**C++ 把"对象生命周期"和"资源管理"绑定**（RAII），析构是程序结构的一部分，确定且可推理。代价是程序员要正确管理所有权（智能指针、move 语义等）。

**Java 把"内存生命周期"交给 GC**，换取内存安全和开发效率，但因此**故意放弃了确定性析构**——这也是为什么 Java 中"非内存资源"（文件句柄、socket、锁）必须用 `try-with-resources` 显式管理，因为 GC 只擅长回收内存，对其他资源既不及时也不可靠。

| 共同点 | 说明 |
|--------|------|
| 用户能写代码的任何时刻，类型指针都是"已设置好"的状态 | 你不需要、也无法直接管理类型指针 |
| "对象一旦分配，类型就确定" | 只是 Java 全程不变、C++ 分层升降的差别，不是"存不存在"的差别 |
