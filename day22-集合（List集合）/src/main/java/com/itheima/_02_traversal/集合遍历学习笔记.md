## 一、forEach 在做什么

`coll.forEach(...)` 是集合的遍历方法。它的作用是：**把集合里的每个元素挨个取出来，交给你提供的"处理逻辑"去处理。**

- `forEach` 负责"循环"。
- 但**具体怎么处理每个元素**它不知道，需要你告诉它。
- 这个"处理逻辑"就是一个 `Consumer` 对象。

forEach 内部可以理解成这样的伪代码：

```java
for (String s : 集合) {
    consumer.accept(s);   // 每个元素都调用一次你写的 accept
}
```

---

## 二、Consumer 和 accept 是什么

`Consumer`（消费者）是 Java 提供的**函数式接口**，里面只有一个抽象方法 `accept`：

```java
public interface Consumer<T> {
    void accept(T t);   // 接收一个元素 t，对它做点事情（没有返回值）
}
```

- **`accept` 就是"处理一个元素"的方法**，直译是"接受"——"我接收一个元素，然后处理它"。
- `forEach` 每拿到一个元素，就调用一次你的 `accept(元素)`。
- 你写在 `accept` 方法体里的代码，就是对每个元素要做的事。

### accept 必须被实现（重写）

`accept` 是抽象方法（没有方法体），**必须有人给它写出具体实现**，否则 forEach 没代码可执行。
重点：**三种写法都重写了 accept，只是形式不同。**

| 写法 | 你看得到 `accept` 吗 | 重写了吗 |
|---|---|---|
| 匿名内部类 | 看得到（手写） | ✅ |
| Lambda | 看不到（编译器补） | ✅ |
| 方法引用 | 看不到（借用现成方法） | ✅ |

> Lambda 之所以能省，是因为 `Consumer` 是**函数式接口**（只有一个抽象方法）。
> 如果接口有两个以上抽象方法，编译器不知道你要实现哪个，就不能用 Lambda，只能用匿名内部类。

## 三、forEach 的三种写法（同一件事，逐步简化）

### ① 匿名内部类（最完整、最啰嗦）

```java
coll.forEach(new Consumer<String>() {
    @Override
    public void accept(String s) {
        System.out.println("[匿名内部类] " + s);
    }
});
```

你手动 new 了一个 `Consumer`，并亲手实现 `accept`。`s` 就是 forEach 每次传进来的元素。

### ② Lambda（简化）

```java
coll.forEach(s -> System.out.println("[Lambda] " + s));
```

编译器知道 `s -> ...` 就是在实现 `accept`，省掉了模板代码。

| 匿名内部类 | Lambda |
|---|---|
| `public void accept(String s) {` | `s ->` |
| 方法体 `{ ... }` | 箭头右边的代码 |

### ③ 方法引用（再简化）

```java
coll.forEach(System.out::println);
```

当 Lambda 只是把参数原样传给另一个方法时，可以用 `::` 缩写。
`System.out::println` 等价于 `s -> System.out.println(s)`。

## 四、三种遍历方式的区别

### ① 普通 for —— 靠"索引"遍历

```java
for (int i = 0; i < list.size(); i++) {
    System.out.println(i + " -> " + list.get(i));
}
```

- 有计数器 `i`，通过 `list.get(i)` 按下标取元素。
- **优点**：能拿到索引 `i`。需要"第几个"、倒序、跳跃遍历时必须用它。
- **缺点**：写法啰嗦；`LinkedList` 用 `get(i)` 很慢（每次从头数）。

### ② 增强 for —— 只关心"元素本身"

```java
for (String s : list) {
    System.out.println(s);
}
```

- 没有索引，`s` 直接就是元素。
- **优点**：简洁、不会越界、对任何集合都高效（底层用 Iterator 实现）。
- **缺点**：拿不到索引；遍历中不能增删元素，否则抛 `ConcurrentModificationException`。

### ③ Lambda forEach —— 把处理逻辑交给集合

```java
list.forEach(System.out::println);
```

- **优点**：最简洁，函数式风格，适合配合 Stream。
- **缺点**：拿不到索引；遍历中不能增删元素；Lambda 里不能修改外部局部变量。

### 对比表

| | 能拿到索引 | 能边遍历边增删 | 简洁度 | 适用场景 |
|---|---|---|---|---|
| 普通 for | ✅ | 删除可以（但要小心 i）| 一般 | 需要下标、倒序、跳跃遍历 |
| 增强 for | ❌ | ❌ 会报错 | 高 | 只读遍历，最常用 |
| Lambda forEach | ❌ | ❌ 会报错 | 最高 | 只读、函数式、配合 Stream |

---

## 五、实战选择口诀

- **只是想挨个看一遍** → 增强 for 或 forEach（优先这俩）
- **需要知道"第几个"，或倒序、跳跃遍历** → 普通 for
- **边遍历边删一个元素** → `Iterator.remove()` 或 `list.removeIf(...)`
- **边遍历边添加元素** → 必须用 `ListIterator`

## 六、常见的坑：遍历中删除元素

```java
// ❌ 增强 for 中删除 —— 抛 ConcurrentModificationException
for (String s : list) {
    if (s.equals("bbb")) {
        list.remove(s);
    }
}
```

原因：增强 for 底层用 `Iterator`，你绕过它直接 `list.remove`，迭代器发现集合被"偷偷改了"就报错。

正确做法：

```java
// ✅ 用迭代器自己的 remove
Iterator<String> it = list.iterator();
while (it.hasNext()) {
    if (it.next().equals("bbb")) {
        it.remove();
    }
}

// ✅ 或者更简洁
list.removeIf(s -> s.equals("bbb"));
```

**一句话：只读用增强 for / forEach，要改结构用 Iterator / ListIterator / removeIf。**
